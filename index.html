01 / 领域演进

三个发展阶段

2020 — 2022

阶段一：静态 NL2SQL

Parser→SQL 的单一模型路径，高度依赖人工标注数据。学术准确率看起来不错，一旦碰到真实数据库就原形毕露——字段名稍有变化、表结构稍复杂，模型就失效。

Spider 1.0 基线 T5 / BART fine-tuning 无上下文理解
2022 — 2024

阶段二：LLM + Prompt Engineering

GPT-4 的出现让 few-shot + CoT 成为主流范式。RAG 开始引入以增强上下文理解。Spider 1.0 准确率突破 85%，但这只是学术的繁荣——真实企业数据库场景的鸿沟开始显现。

DIN-SQL DAIL-SQL MAC-SQL GPT-4 few-shot + CoT
2024 — 至今

阶段三：Compound AI / Multi-Agent

单一 LLM 遇到了天花板。业界转向多专用 Agent 协作——意图解析、Schema 链接、规划、代码生成、执行验证各司其职。目标是 L4 自主化：端到端完成整个数据工程项目，人类只做决策验收。

Databricks Genie Vanna v2 DS-STAR AutoKaggle L4 Autonomy 目标

语义层的关键作用

Semantic Layer — 防幻觉核心

语义层（Metric Views）预先定义"销售额如何计算、哪些订单算已完成"等业务逻辑。LLM 从语义层读取确定性的规则，而不是自行推断，将准确率从 40–50% 直接提升到 85–90%。

// 无语义层 — LLM 自行推断
40 – 50%

LLM 基于常识猜测 revenue 计算方式 → 度量不一致，幻觉频发

// 有 Metric Views — 确定性逻辑
85 – 90%

从预定义语义层读取度量逻辑 → 单一事实来源，跨工具一致

03 / 研究方向

三条主线

论文 / 系统 发表 核心贡献
MAC-SQL EMNLP 2024 2024 多 Agent 协作：Decomposer + Selector + Refiner 三层分工
DAIL-SQL VLDB 2024 2024 Few-shot 示例选择策略，Spider 1.0 SOTA，奠定 prompt 工程范式
LinkAlign EMNLP 2025 2025 大规模 Schema Linking，query rewriting + response filtering 减少错误
DBCopilot 2025 2025 Schema Routing，支持千表级超大数据库的 NL 查询
E-SQL 2024 2024 Direct Schema 链接，减少中间步骤，降低 latency
Schema Linking（大型 DB） — 10,000+ 张表如何快速定位相关表？两阶段 Recall（向量粗筛 + LLM 精排）是当前主流方案
跨 SQL 方言 — BigQuery ARRAY_AGG vs Snowflake LISTAGG vs MySQL GROUP_CONCAT，方言异构是真实部署的拦路虎
多轮对话一致性 — "它的上个月" 中 "它" 指什么？上下文依赖的消歧至今缺少通用解法

AutoKaggle

5 个专用 Agent 流水线：
Reader → Planner → Developer → Reviewer → Summarizer

面向 Kaggle 数据竞赛，自动完成特征工程到模型训练。2024 年发表，目前是 multi-agent EDA 的基准参照。

DS-STAR

核心创新：引入 LLM-as-Judge 验证层，不仅依赖执行反馈。

在 DABStep（450 任务, 7 个数据文件）和 KramaBench（6 领域, 1556 文件）上达到 SOTA，2025 年发表。

ADP-MA

Autonomous Data Processing via Meta-Agents。

动态确定工作流阶段（vs AutoKaggle 预定义顺序），更灵活，支持超出预测建模的广义数据处理任务。2026 年初发表。

自主化等级

研究目标已明确指向 L4 自主化：系统可自编排多阶段 pipeline，人类只提目标不管过程。当前大部分产品在 L1–L3 之间（辅助单个孤立任务）。

LightVA

生成 Python / SQL → 沙箱执行 → 验证结果 → 自我修正循环。强制通过执行反馈来保证正确性，而不是单次生成祈祷能跑。

Jupybara

规划 Agent + 审查 Agent 分离设计。由独立审查 Agent 对生成代码进行迭代验证，降低未检测到的幻觉概率。

Google BigQuery Code Interpreter

2025 年 GA，集成在 BigQuery Studio。NL → Python → 图表，面向业务用户，复杂分析问题（如"做客户分群"）直接生成可执行 Python 代码。

意图理解失误 — LLM 生成的代码逻辑上正确，但功能上不符合用户真实需求
包名 / API 幻觉 — 模型发明不存在的 Python package 或数据库字段名，沙箱执行是目前唯一可靠的检测手段

05 / 关键挑战

准确率断崖

这是数据分析 Agent 领域最核心的现实困境：模型在学术基准上表现出色，在真实企业工作流中却几乎失效。

⚠️

Spider 2.0 包含来自 Salesforce、Google Analytics 等真实工业项目的任务，涉及跨数据库方言、超大 Schema、复杂多步骤业务推理。GPT-4o 在此的准确率仅 10.1%，与 Spider 1.0 的 86.6% 相比，断崖式下跌暴露了当前方案的核心局限。

🗄️

Schema Linking（大型 DB）

10,000+ 张表的企业数据库，向量检索召回率有限。"Lost in the Middle" 问题：正确 Schema 在上下文里但被模型忽略。

👻

幻觉（列名 / 函数名）

LLM 发明不存在的字段名或 API，静默失败（查询返回 0 行而非报错），比语法错误更难排查。

💬

多轮对话一致性

"它的上个月增长率" 中 "它" 指代什么？Chat history 过滤策略是当前主要方案，但缺少通用解法。

🔀

SQL 方言异构

BigQuery / Snowflake / Spark SQL / MySQL 方言差异大，同一查询逻辑需不同语法。方言感知 fine-tuning 仍是未解难题。

06 / 趋势判断

接下来会发生什么

近期（1–2 年）确定性高
01
语义层标准化

Databricks Unity Catalog Business Semantics 开源、dbt MetricFlow 普及，各厂商将收敛到统一语义标准

02
MCP 协议普及

Agent 通过 MCP 连接 Jira / Confluence / GitHub，数据分析 Agent 成为跨系统 Orchestrator

03
专用 SQL 小模型

SQLCoder 系列证明 7B fine-tune 可超越 GPT-4，私有化部署和成本敏感场景的需求驱动

04
Agentic EDA 成熟化

从 "帮你写 SQL" 进化到 "帮你完成整个分析项目"，AutoKaggle 范式向企业场景迁移

中期（2–4 年）高概率
01
L4 自主化

全流程自主（提问 → EDA → 建模 → 报告），人类只做决策验收。当前最优系统在 L2–L3

02
多模态融合

图表理解 + 表格推理融合，用户截一张 Excel 截图就能触发数据查询分析

03
因果推理

从 "描述性分析"（发生了什么）向 "因果分析"（为什么发生、如何干预）进化

07 / 核心结论

现状一句话

"数据分析 Agent 当前处于
'学术任务已基本解决，真实企业场景仍是未解难题'
的阶段。"
🎯
核心矛盾：学术基准 90%+ ≠ 真实场景可用。Spider 2.0 用 10.1% 的准确率告诉我们，增大 LLM 不是答案。
🔑
真正的突破口在 Schema Linking + 语义理解，而不是更大的基础模型。让 LLM 理解企业数据的语义，比让它"更聪明"更重要。
⚡
语义层（Metric Views）是目前最有效的工程化解法：40% → 87% 的准确率提升，靠的是预定义确定性逻辑，不是更大的模型。
