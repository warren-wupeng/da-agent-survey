01 / 领域演进

Three Development Stages

2020 — 2022

Stage 1: Static NL2SQL

A single-model pipeline from Parser→SQL, heavily reliant on manually annotated data. Academic accuracy looks decent, but falls apart when faced with real databases — slight changes in field names or slightly more complex table structures cause the model to fail.

Spider 1.0 Baseline T5 / BART fine-tuning No Contextual Understanding
2022 — 2024

Stage 2: LLM + Prompt Engineering

The emergence of GPT-4 made few-shot + CoT the dominant paradigm. RAG was introduced to enhance contextual understanding. Spider 1.0 accuracy broke through 85%, but this was merely academic prosperity — the gap with real enterprise database scenarios began to emerge.

DIN-SQL DAIL-SQL MAC-SQL GPT-4 few-shot + CoT
2024 — 至今

Stage 3: Compound AI / Multi-Agent

Single LLMs hit a ceiling. The industry shifted toward collaboration among multiple specialized Agents — intent parsing, Schema linking, planning, code generation, and execution validation each handling their own responsibilities. The goal is L4 Autonomy: completing entire data engineering projects end-to-end, with humans only making final decisions.

Databricks Genie Vanna v2 DS-STAR AutoKaggle L4 Autonomy Goal

语义层的关键作用

Semantic Layer — Anti-Hallucination Core

The Semantic Layer (Metric Views) pre-defines business logic such as "how revenue is calculated" and "which orders count as completed". The LLM reads deterministic rules from the semantic layer instead of inferring them on its own, directly improving accuracy from 40–50% to 85–90%.

// 无语义层 — LLM 自行推断
40 – 50%

LLM guesses revenue calculation based on common sense → inconsistent metrics, frequent hallucinations

// 有 Metric Views — 确定性逻辑
85 – 90%

Reads metric logic from a predefined semantic layer → single source of truth, consistent across tools

03 / 研究方向

Three Main Tracks

Paper / System Published Core Contribution
MAC-SQL EMNLP 2024 2024 Multi-Agent collaboration: three-tier division of Decomposer + Selector + Refiner
DAIL-SQL VLDB 2024 2024 Few-shot example selection strategy, Spider 1.0 SOTA, establishing the prompt engineering paradigm
LinkAlign EMNLP 2025 2025 Large-scale Schema Linking, query rewriting + response filtering to reduce errors
DBCopilot 2025 2025 Schema Routing, supporting NL queries on ultra-large databases with thousands of tables
E-SQL 2024 2024 Direct Schema linking, reducing intermediate steps and lowering latency
Schema Linking (Large DB) — How to quickly locate relevant tables among 10,000+ tables? Two-stage Recall (vector coarse filtering + LLM re-ranking) is the current mainstream approach
Cross-SQL Dialect — BigQuery ARRAY_AGG vs Snowflake LISTAGG vs MySQL GROUP_CONCAT, dialect heterogeneity is a major obstacle in real-world deployment
Multi-turn Conversation Consistency — What does "it" refer to in "its last month"? Context-dependent disambiguation still lacks a universal solution

AutoKaggle

5-agent specialized pipeline:
Reader → Planner → Developer → Reviewer → Summarizer

Targeting Kaggle data competitions, automating everything from feature engineering to model training. Published in 2024, currently the benchmark reference for multi-agent EDA.

DS-STAR

Core innovation: introduces an LLM-as-Judge validation layer, not solely relying on execution feedback.

Achieves SOTA on DABStep (450 tasks, 7 data files) and KramaBench (6 domains, 1556 files), published in 2025.

ADP-MA

Autonomous Data Processing via Meta-Agents.

Dynamically determines workflow stages (vs AutoKaggle's predefined order), more flexible, supporting generalized data processing tasks beyond predictive modeling. Published in early 2026.

自主化等级

Research goals are now clearly targeting L4 autonomy: systems can self-orchestrate multi-stage pipelines, with humans only providing goals without managing the process. Most current products are between L1–L3 (assisting individual isolated tasks).

LightVA

Generate Python / SQL → sandbox execution → validate results → self-correction loop. Correctness is enforced through execution feedback rather than relying on a single generation and hoping it runs.

Jupybara

Separated design of Planning Agent + Review Agent. An independent review Agent iteratively validates generated code, reducing the probability of undetected hallucinations.

Google BigQuery Code Interpreter

GA in 2025, integrated in BigQuery Studio. NL → Python → charts, targeting business users, complex analytical questions (e.g., "perform customer segmentation") directly generate executable Python code.

Intent misunderstanding — LLM-generated code is logically correct but functionally does not meet the user's actual needs
Package name / API hallucination — the model invents non-existent Python packages or database field names; sandbox execution is currently the only reliable detection method

05 / 关键挑战

Accuracy Cliff

This is the most critical real-world challenge in the data analysis Agent domain: models perform well on academic benchmarks but nearly fail in real enterprise workflows.

⚠️

Spider 2.0 contains tasks from real industrial projects such as Salesforce and Google Analytics, involving cross-database dialects, extremely large schemas, and complex multi-step business reasoning. GPT-4o achieves only 10.1% accuracy here, compared to 86.6% on Spider 1.0 — a cliff-like drop that exposes the core limitations of current approaches.

🗄️

Schema Linking (Large DB)

Enterprise databases with 10,000+ tables have limited vector retrieval recall. "Lost in the Middle" problem: the correct schema is in the context but gets ignored by the model.

👻

Hallucination (column names / function names)

LLM invents non-existent field names or APIs, failing silently (query returns 0 rows instead of an error), making it harder to debug than syntax errors.

💬

Multi-turn conversation consistency

In "its growth rate last month", what does "its" refer to? Chat history filtering strategies are the current primary approach, but a universal solution is lacking.

🔀

SQL dialect heterogeneity

BigQuery / Snowflake / Spark SQL / MySQL dialects differ significantly; the same query logic requires different syntax. Dialect-aware fine-tuning remains an unsolved challenge.

06 / 趋势判断

What's Coming Next

近期（1–2 年）确定性高
01
Semantic layer standardization

Databricks Unity Catalog Business Semantics goes open source, dbt MetricFlow becomes widespread, and vendors will converge on a unified semantic standard

02
MCP protocol adoption

Agents connect to Jira / Confluence / GitHub via MCP, and data analysis Agents become cross-system Orchestrators

03
Specialized small SQL models

The SQLCoder series proves that 7B fine-tuning can surpass GPT-4, driven by demand for private deployment and cost-sensitive scenarios

04
Agentic EDA maturation

Evolving from "helping you write SQL" to "helping you complete entire analysis projects", with the AutoKaggle paradigm migrating to enterprise scenarios

中期（2–4 年）高概率
01
L4 Autonomy

Fully autonomous end-to-end pipeline (Question → EDA → Modeling → Report), with humans only responsible for decision validation. Current best systems are at L2–L3

02
Multimodal Fusion

Chart understanding + table reasoning fusion, users can trigger data query analysis by simply taking a screenshot of an Excel file

03
Causal Reasoning

Evolving from "descriptive analysis" (what happened) to "causal analysis" (why it happened, how to intervene)

07 / 核心结论

Current Status in a nutshell

"数据分析 Agent 当前处于
'Academic tasks are largely solved, real enterprise scenarios remain an unsolved challenge'
的阶段。"
🎯
Core Contradiction：学术基准 90%+ ≠ 真实场景可用。Spider 2.0 用 10.1% 的准确率告诉我们，增大 LLM 不是答案。
🔑
The real breakthrough lies in Schema Linking + semantic understanding，而不是更大的基础模型。让 LLM 理解企业数据的语义，比让它"更聪明"更重要。
⚡
语义层（Metric Views）是目前Most Effective Engineering Solutions：40% → 87% 的准确率提升，靠的是预定义确定性逻辑，不是更大的模型。
